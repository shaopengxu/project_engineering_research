## 探索项目开发的最佳模式

### 话题：需求拆分、多角色Agent、文档驱动、测试驱动、角色分离、上下文管理

---

### Q: 探索项目开发的最佳模式。需求拆分模块，多角色agent，文档驱动，测试驱动，怎么做角色分离，是否需要一个项目经理的agent，怎么考虑上下文的容量，什么时候reset会话

### A:

## 1. 需求拆分：分层模块化

```
需求文档 (PRD)
  └── Epic (大功能)
        └── Feature (可独立交付的模块)
              └── Task (单次会话可完成的工作单元)
```

关键原则：每个 Task 应该是一个 agent 在单次会话中能完成的粒度。判断标准：
- 涉及的文件 < 15 个
- 改动行数 < 500 行
- 能用一句话描述清楚目标

## 2. 多角色 Agent 分离

推荐 4 种核心角色：

| 角色 | 职责 | 触发时机 |
|------|------|----------|
| Architect | 设计模块边界、API 接口、数据结构 | 项目初始 / 新 Feature 启动 |
| Implementer | 写代码，只关注当前 Task | 每个 Task |
| Reviewer | 审查代码质量、安全性、一致性 | PR / 代码完成后 |
| Tester | 写测试、跑测试、验证边界条件 | 实现完成后 |

### 是否需要 "项目经理" Agent？

不需要一个持久运行的 PM agent。原因：
- Agent 没有跨会话的持久状态，PM 角色需要持续跟踪进度
- 更好的方案是用文档 + 人来承担 PM 职责

替代方案：
```
CLAUDE.md          — 项目约定（所有 agent 都读）
docs/plan.md       — 当前迭代计划（人维护）
docs/task-board.md — 任务状态（人 + agent 共同更新）
```

## 3. 文档驱动开发

核心思路：文档是 agent 之间的通信协议。

```
docs/
├── architecture.md    # Architect 产出，Implementer 消费
├── api-contracts.md   # 接口定义，前后端 agent 共享
├── task-board.md      # 任务状态追踪
└── decisions/
    └── 001-auth.md    # ADR (Architecture Decision Records)
```

工作流：
1. 人写 PRD → Architect agent 产出 architecture.md + api-contracts.md
2. 人拆 Task → 写入 task-board.md
3. Implementer agent 开始前先读 CLAUDE.md + 相关文档
4. 完成后更新 task-board.md

## 4. 测试驱动开发

推荐 "测试先行" 的 agent 工作流：
Step 1: Architect agent → 定义接口 + 写接口测试骨架
Step 2: Tester agent   → 补全测试用例（边界、异常）
Step 3: Implementer    → 跑测试 → 写代码 → 测试通过
Step 4: Reviewer agent → 审查

## 5. 上下文容量管理

原则：每个会话只装它需要的上下文

策略：
- CLAUDE.md 分层：根目录放全局规则，子目录放模块特定规则
- 文档 < 代码：让 agent 读文档理解架构，而非遍历所有代码
- 单一职责会话：一个会话只做一个 Task，不要跨 Task 复用
- 输出为输入：前一个 agent 的产出文档作为下一个的输入

## 6. 什么时候 Reset 会话

必须 Reset：
- Task 完成时
- 上下文接近上限时
- 方向性转变时
- agent 犯重复错误时

不需要 Reset：
- 同一个 Task 内的迭代修改
- 测试失败后的修复循环
- 代码审查反馈的修改

## 7. 推荐的实战流程

```
人: 写 PRD
 ↓
会话1 [Architect]: 读 PRD → 产出 architecture.md, api-contracts.md
 ↓
人: 拆分 Tasks → task-board.md
 ↓
会话2 [Tester]: 读 api-contracts → 写测试骨架
 ↓
会话3 [Implementer]: 读文档 + 测试 → 实现模块A → 测试通过
会话4 [Implementer]: 读文档 + 测试 → 实现模块B → 测试通过  (可并行)
 ↓
会话5 [Reviewer]: 审查所有改动
 ↓
人: 合并，更新 task-board
```

### Q: CLAUDE.md 应该写哪些东西

### A:

CLAUDE.md 是每次会话自动加载的项目指令文件，核心原则：只写 agent 无法从代码中推断出的信息。

## 结构模板

```markdown
# Project: {项目名}

## 一句话描述
{这个项目是什么，解决什么问题}

## Tech Stack
- Language: TypeScript 5.x
- Runtime: Node 20
- Framework: Next.js 14 (App Router)
- DB: PostgreSQL + Prisma ORM
- Test: Vitest + Playwright

## 常用命令
- `npm run dev` — 启动开发服务器
- `npm test` — 跑全部测试
- `npm run test:unit -- path/to/file` — 跑单个文件测试
- `npm run lint` — lint 检查
- `npm run build` — 构建

## 代码规范
- 使用函数式组件，不用 class 组件
- 错误处理用 Result 模式，不用 try-catch
- 文件命名: kebab-case
- 组件命名: PascalCase

## 项目结构
src/
├── app/          # Next.js 路由
├── components/   # 共享 UI 组件
├── lib/          # 业务逻辑
├── services/     # 外部 API 调用
└── types/        # 共享类型定义

## 架构约定
- API 路由统一返回 { data, error } 格式
- 数据库操作只在 services/ 层，不在组件中直接调用
- 所有环境变量在 lib/env.ts 中集中校验

## 不要做的事
- 不要修改 migrations/ 中已有的迁移文件
- 不要在组件中直接用 fetch，用 services/ 封装
- 不要 commit .env 文件
```

每个区块的作用：
- 一句话描述：Agent 理解业务上下文
- Tech Stack：避免猜错版本/框架
- 常用命令：Agent 能直接跑测试/构建
- 代码规范：产出风格一致
- 项目结构：知道代码放哪
- 架构约定：遵守设计决策
- 不要做的事：防止高代价错误

关键原则：
1. 短。控制在 100-200 行以内
2. 只写不可推断的
3. 写约束，不写教程
4. 分层放置（子目录可以有自己的 CLAUDE.md）
5. 持续维护

### Q: 如果在同一个会话问大模型，大模型实际上还保留之前的上下文对不对，那每次会话都还加载CLAUDE.md 那是不是重复了

### A:

在同一个会话内，CLAUDE.md 只加载一次（会话开始时）。不存在重复加载的问题。

```
会话开始
  ├── 自动加载 CLAUDE.md → 写入上下文（一次性）
  ├── 你的第1条消息
  ├── Agent 回复
  ├── 你的第2条消息   ← CLAUDE.md 的内容还在上下文里
  ├── ...             ← 一直在，直到会话结束
  └── 上下文接近上限时 → 系统自动压缩早期消息（但 CLAUDE.md 会被保留）
```

CLAUDE.md 真正的价值在于跨会话 — 没有 CLAUDE.md 的话，每次新会话都得手动重复告诉 agent 项目规则。CLAUDE.md 本质上是跨会话的共享记忆。

---

### Q: 如果让agent每次在回复时都简短的记录处理的问题和解决方式，以便于之后参考，是否有必要

### A:

大多数情况下没必要，甚至有害。

原因：
1. 代码本身就是最好的记录（git log, git blame, git diff）
2. 浪费上下文（每轮100 tokens，20轮就是2000 tokens）
3. 笔记会过时，过时的笔记比没有笔记更危险
4. 同一会话内不需要，上下文还在

真正值得记录的：踩了坑，且这个坑从代码中看不出来。

推荐替代方案：
- 代码层面：git commit message 写清楚 why
- 项目层面：遇到非显而易见的坑 → 写进 CLAUDE.md
- 架构层面：重大决策 → 写 ADR

---

### Q: 那每次修改都要记录git，无论是文档还是代码，这个是不是应该写到全局记忆里

### A:

是的。已记录到全局记忆：每次改动（无论代码还是文档）都应该 commit。

---

### Q: 那是不是应该加一个这样的全局记忆：每次一个新的项目开始提醒用户应该记录git，并且根据需求建议使用怎样的git策略

### A:

已记录。触发条件改为：只在检测到未初始化的 git repo（没有 .git 目录）时触发，已初始化的项目不会重复提醒。

---

### Q: memory也是每次会话开始的时候会加载吗

### A:

是的，和 CLAUDE.md 一样，MEMORY.md 索引文件在会话开始时自动加载。

但有区别：
- CLAUDE.md → 内容直接加载
- MEMORY.md → 只加载索引（一行摘要），具体 .md 文件按需读取

加载层级：
- 全局：~/.claude/memory/MEMORY.md（所有项目）
- 项目级：.claude/projects/{项目}/memory/MEMORY.md（当前项目）

两层都会加载，不冲突。全局放通用习惯，项目级放项目特定上下文。

---

### Q: 但我加的这样提醒实际上只需要在项目开始的时候提醒，在项目过程中新增会话并不需要这样的提醒，是否有只针对项目初始化的触发机制

### A:

没有内置的"仅项目初始化时触发"机制。但可以把触发条件写进记忆内容里，让 agent 自己判断：只在检测到没有 .git 目录时触发，已初始化就跳过。

### Q: hook是什么呢

### A:

Hooks 是 Claude Code 的功能：在特定事件发生时自动执行 shell 命令。

支持的触发点：
- PreToolUse：工具执行之前
- PostToolUse：工具执行之后
- Notification：Agent 发通知时
- Stop：Agent 完成回复时
- SubagentStop：子 Agent 完成时

和 Memory/CLAUDE.md 的区别：
- CLAUDE.md / Memory：文字指令，Agent 自己判断遵守（可能忽略）
- Hook：Shell 脚本，系统强制执行，100% 触发

---

### Q: 那如果让hook做commit 它能根据功能拼装出commit时的message吗

### A:

不能。Hook 只是执行 shell 命令，它不知道 agent 做了什么。

更好的方案：
- 在 CLAUDE.md 里写 commit message 格式规则，让 agent 写有意义的 message
- 用 hook 做提醒（检测未提交改动），让 agent 做执行

Hook 负责"不遗漏"，Agent 负责"写好 message"。

### Q: 如果我在同一个目录开启多个claude作为不同角色去执行，会有什么问题吗

### A:

主要问题是文件冲突。

风险：
- 编辑同一文件：互相覆盖（高风险）
- 同时 git commit：冲突或交叉提交（高风险）
- 同时跑 dev server：端口冲突（中风险）
- 同时装依赖：lock file 冲突（中风险）
- 读同一文件 / 编辑不同文件：无问题

推荐方案：Git Worktree
- 每个 agent 有自己的文件副本和独立分支
- 完全不会冲突
- 最后通过 git merge 合并

```bash
git checkout -b feature-a
git worktree add ../worktree-feature-b feature-b
git worktree add ../worktree-feature-c feature-c
```

### Q: 回到hook，hook一般用在哪些场景

### A:

常见场景：
1. 代码质量守护：PostToolUse + Write|Edit → 自动 lint
2. 未提交改动提醒：Stop → 检测 git status
3. 阻止危险操作：PreToolUse + Bash → 拦截 rm -rf 等
4. 自动跑测试：PostToolUse + Write|Edit → 改了 src 下文件自动跑测试
5. 通知：Stop → macOS 弹通知
6. 审计日志：PreToolUse / PostToolUse → 记录所有操作

核心判断标准：如果一件事你不希望 agent "可能忘记"，就用 Hook 而不是 CLAUDE.md。

### Q: 审计日志什么时候需要

### A:

需要的场景：
1. 多人团队共用 Claude Code — 追踪谁的 agent 做了什么
2. 合规/安全审计 — 金融、医疗、政府项目
3. 调试 Agent 行为 — 理解 agent 为什么做出某个决策
4. 成本分析 — 发现低效的 prompt 或配置

个人项目研究阶段不需要。

### Q: 我之前尝试过让agent做项目经理去管其他agent进行项目开发，效果还可以，因为知道当在做什么，下一步要做什么本身也需要花费token去编排，单独的项目经理agent可以减少做具体任务的agent的上下文开销

### A:

你说的有道理，修正之前的观点。

PM agent 的价值：上下文的"关注点分离"。

```
没有 PM agent:
  每个 agent 都要花上下文在"编排"上（理解计划、判断进度、决定下一步）

有 PM agent:
  PM agent 花 token 在编排上
  执行 agent 把上下文全部用在任务本身
```

要让 PM agent 有效，关键是用 task-board.md 做"外部记忆"，每次新会话通过读文件恢复状态。

修正后的建议：
- 小项目：不需要 PM agent
- 中等项目：看情况
- 大项目/多 agent 并行：需要，编排开销值得独立出来

### Q: 说说对模块的划分，怎么合理的划分模块

### A:

## 两个维度

- 业务维度（垂直切分）：用户管理、订单管理、支付系统
- 架构维度（水平切分）：UI层、业务逻辑层、数据访问层

推荐：业务优先，架构内嵌。

## 划分判断标准

问三个问题：
1. 能否独立开发？
2. 能否独立测试？
3. 能否独立理解？

## 模块间依赖的处理原则

1. 依赖方向单向（不能有循环）
2. 依赖接口，不依赖实现
3. 共享的东西往下沉（shared/infra）
4. 用事件解耦强关联

模块依赖图就是 agent 的并行调度图。

### Q: user不依赖order，order依赖user，但如果要查某个用户的所有订单，从业务角度讲它属于user，但从代码实现上讲，它应该属于order。怎么理解这种割裂

### A:

其实不矛盾。区分"数据归属"和"功能入口"。

URL 和页面属于 user 的领域，但执行逻辑属于 order：
```typescript
// user.controller.ts — 入口在 user
router.get('/users/:id/orders', async (req) => {
  return orderService.getByUserId(req.params.id)
})
```

如果双向调用都存在，可以引入聚合层。

判断"放哪"的方法：数据从哪来？数据在哪个表 → 就属于哪个模块。业务上的"属于谁"是 UI 和路由的问题，代码上的"放在哪"跟着数据走。

---

### Q: 聚合层会变得庞大怎么办

### A:

关键转变：没有"聚合层"这个概念。跨模块的编排本身就是一个业务模块。

```
"下单结算" 涉及 order + payment + inventory
→ checkout 就是一个独立的业务模块，不是"聚合"
```

判断标准：
- 有业务规则 → 它是一个业务模块
- 没有业务规则，只是拼数据 → 放在 controller 里直接调

过早提取抽象层是复杂度的来源。三行重复代码好过一个错误的抽象。

---

### Q: 产品经理用AI开发遇到问题怎么办，有没有万能prompt

### A:

没有万能 prompt。核心问题是反馈回路断了 — PM 无法做 AI 的"质检员"。

解法：
1. 测试是非技术人员的"眼睛" — PM 确认测试用例是否符合需求，AI 写代码让测试通过
2. 小步验证 — 每一步让 PM 实际操作验证
3. 强制检查点 — CLAUDE.md 里写规则（遇到技术问题不要自己绕过，先向用户解释）
4. 错误方向检测 — 每完成一个功能对比原始需求

天花板：复杂架构决策、性能优化、安全问题仍需技术人员。

务实建议：项目开始时让技术顾问定好架构和 CLAUDE.md，之后 PM 在框架内推进。

---

### Q: 产品经理遇到AI反复修改都无法解决的问题怎么办

### A:

AI 反复修不好通常三个原因：需求不清晰、方向错了、超出能力。

处理流程：
1. 超过 3 轮修改还没解决 → 停
2. 开新会话，只描述现象和期望（不带修改历史）
3. 新会话也解决不了 → 让 AI 生成问题报告，找技术人员

可写进 CLAUDE.md 的防护机制：
- 修改超过 2 次未解决 → 停止修改，分析根本原因
- 给出不同方案或建议寻求技术帮助
- 禁止在同一方向尝试超过 3 次

### Q: 拆分模块是在需求就开始拆分还是在技术设计阶段再拆分

### A:

两个阶段都要拆，但拆的东西不一样。

需求阶段：拆业务边界（PM 就能做）
技术设计阶段：拆代码边界（需要技术判断）

维护一张映射表很重要 — PM 说"下单有 bug"，技术能定位到是哪个代码模块的问题。

流程：
1. PM 拆业务模块
2. 技术拆代码模块 + 映射表
3. PM 确认映射关系
4. 按技术模块分配给 agent 开发，按业务模块让 PM 验收

---

### Q: 开发代码的时候还需要参考PRD吗

### A:

需要，但不是直接读 PRD。

正确链路：PRD → 技术设计文档 → Task 描述 → Agent 写代码

Agent 真正需要的输入是明确聚焦的 Task 描述，不是完整 PRD。
PRD 留着备查，在边界情况决策和验收时使用。

---

### Q: 需求分模块，需要拆到多个文件吗

### A:

看项目大小：
- PRD < 500 行 → 一个文件，用标题分模块
- PRD 500-1500 行 → 看情况
- PRD > 1500 行 → 必须拆

拆文件的唯一理由：agent 的上下文装不下。不要为了"看起来专业"而拆。
