# 中型项目 vs 小型项目：规模与复杂度变化分析

> 本文档只分析问题，不给解决方案。目标是精确识别小项目流程在哪些点上会"崩溃"。

---

## 1. 模块数量和关系

### 1.1 量变

| 维度 | 小型项目 | 中型项目 |
|------|---------|---------|
| 业务模块数 | 1-3 | 4-10 |
| 技术模块数 | 2-5 | 8-20（业务模块 + 基础设施 + 共享层膨胀） |
| 数据模型数 | 1-3 张表 | 8-20 张表 |
| 模块间接口数 | 3-8 个 | 30-80 个 |

### 1.2 质变：依赖关系从"链"变成"图"

小型项目的模块依赖是**线性链**或**简单扇出**：

```
小项目典型依赖（bookmark-api 示例）：

shared + infra
  ↑          ↑
bookmark    tag     ← 两个业务模块互不依赖，通过关联表在数据库层交互
```

中型项目的模块依赖是**有向图**，可能出现：

```
中型项目典型依赖（电商后台示例）：

shared + infra
  ↑    ↑    ↑    ↑    ↑
user  product  order  payment  inventory
        ↑        ↑↗     ↑        ↑
        └── order ──┘    │        │
            ↑            │        │
            └── payment ─┘        │
            ↑                     │
            └── inventory ────────┘
                ↑
            checkout（编排模块，依赖 order + payment + inventory）
```

### 1.3 小项目流程在此崩溃的具体点

**崩溃点 1：模块依赖关系无法用文本描述清楚**

小项目的 architecture.md 用简单的 ASCII 箭头图就能表达依赖关系（3-5 行）。中型项目的依赖关系超过 10 条边后，文本图变得难以阅读且容易遗漏。agent 在实现时可能违反依赖约束而不自知。

**崩溃点 2：出现循环依赖的诱惑**

小项目模块少（2-3 个），人工就能保证不循环。中型项目 6+ 模块之间很容易出现 A→B→C→A 的隐性循环，尤其当模块间需要"反向查询"时（如 talk.txt 中讨论的"查用户的订单"问题）。小项目流程没有循环依赖检测机制。

**崩溃点 3：共享层失控**

小项目的 shared/ 准入规则是"被 2 个以上模块使用"。当模块数到达 6-10 个时，几乎所有工具函数都满足"2 个以上模块使用"的条件，shared/ 会迅速膨胀为一个"什么都有"的垃圾抽屉。原先"每个文件 < 100 行"的约束不足以阻止 shared/ 整体变成一个巨大的隐性模块。

**崩溃点 4：多对多关系爆炸**

小项目最多一个多对多关系（bookmark-tag）。中型项目可能有 5+ 个多对多关系（用户-角色、商品-标签、订单-商品、用户-收藏、文章-分类...），每个多对多都意味着一个关联表、一套关联管理逻辑、一个"归属到哪个模块"的设计决策。小项目流程中"关联逻辑放在 URL 前缀所属模块"的简单规则不够用了——当同一个关联两端的模块都有强业务需求操作这个关联时，简单归属规则就会导致要么一端被迫跨模块调用，要么出现逻辑重复。

**崩溃点 5：事件级联和副作用链**

小项目中"删除书签时清除关联"通过数据库 ON DELETE CASCADE 解决，副作用是数据库层面的，透明且可预测。中型项目中，一个操作可能触发跨多个模块的业务级联：

- 删除用户 → 取消该用户的所有未完成订单 → 恢复每个订单锁定的库存 → 触发库存变更通知
- 下单 → 扣减库存 → 创建支付单 → 发送下单通知 → 更新用户积分

这种业务副作用链无法用数据库外键解决，需要显式编排。小项目流程没有处理跨模块业务事件的机制。

**崩溃点 6：共享状态和一致性**

小项目的数据基本上是"各管各的"，偶尔通过关联表交互。中型项目出现共享状态问题：库存数量被订单模块和库存模块同时读写、用户余额被支付模块和积分模块同时操作。小项目流程没有讨论并发访问和数据一致性问题。

## 2. 团队规模

### 2.1 量变

| 维度 | 小型项目 | 中型项目 |
|------|---------|---------|
| 开发人数 | 1-2 人 | 3-6 人 |
| 并行 agent 会话数 | 1-2 个 | 3-8 个 |
| 需要协调的通信通道数 | 1 条（2人之间） | 3-15 条（n*(n-1)/2） |

### 2.2 小项目流程在此崩溃的具体点

**崩溃点 7：task-board.md 成为并发瓶颈**

小项目的任务看板更新规则是"由技术负责人统一更新，避免多人同时编辑冲突"。这隐含假设技术负责人就 1 个人、任务量少到可以手动同步。3-6 人团队中，如果 4 个 agent 同时完成任务需要更新看板，技术负责人变成信息瓶颈。任务状态的实时性丧失——agent 读到的可能是过时的看板状态，导致重复工作或遗漏依赖。

**崩溃点 8：单一 CLAUDE.md 无法承载分模块的差异化规则**

小项目一个 CLAUDE.md 涵盖全部规则（50-60 行）。中型项目不同模块可能有不同的代码规范、测试策略、依赖约束。例如 payment 模块需要额外的安全规范，admin 模块需要特殊的权限检查规则。把所有规则堆在一个 CLAUDE.md 中，要么太长（超过 200 行的有效信息上限），要么 agent 执行其他模块任务时被不相关的规则干扰。

**崩溃点 9：代码风格分裂**

小项目 1-2 个人（或 agent），代码风格天然一致。3-6 个人/agent 并行开发时，即使有 CLAUDE.md 的代码规范，不同 agent 会话对规范的理解和执行会有偏差。错误处理方式、命名风格、注释详细程度、测试编写模式都会逐渐分化。小项目流程依赖"单人一致性"，没有"多人一致性"的保障机制。

**崩溃点 10：Git 工作流从线性变为并行**

小项目的 Git 规则只有 commit message 格式，隐含假设是线性开发（一个任务做完再做下一个）。中型项目 3+ 人并行开发时面临：

- 多分支同时修改同一个共享文件（如 database.ts 加新表、app.ts 注册新路由）
- 合并冲突的发现和解决
- 分支命名和管理策略
- 不同模块进度不一导致的集成风险

小项目流程的 task-board 中虽然为每个 Impl 任务分配了分支（feature/impl-001），但没有合并策略、冲突解决流程、集成测试时机的定义。

**崩溃点 11：Review 流程无法串行执行**

小项目的 review 是"技术负责人在所有测试/实现完成后统一 review"。中型项目如果等所有实现做完再 review，积累的代码量太大（可能几千行），review 质量必然下降。但如果改为逐任务 review，review 的频率和人力成本又大幅上升。小项目流程没有定义增量 review 策略。

## 3. 迭代与变更管理

### 3.1 量变

| 维度 | 小型项目 | 中型项目 |
|------|---------|---------|
| 迭代次数 | 1 次（一口气做完） | 3-8 次（分迭代交付） |
| 需求变更概率 | 低（规模小，需求基本不变） | 高（模块多，业务环境变化影响面大） |

### 3.2 小项目流程在此崩溃的具体点

**崩溃点 12：没有迭代/版本概念**

小项目流程假设项目是一次性完成的：PRD → 架构 → 测试 → 实现 → review → 完成。没有迭代规划、版本划分、渐进式交付的机制。中型项目模块多、复杂度高，需要能分阶段交付可用版本，让用户/PM 尽早验证方向。当前流程的 task-board 是一个"全量任务列表"，没有按迭代组织任务的能力。

**崩溃点 13：需求变更的影响范围无法评估**

小项目的 task-board 有一个"变更场景示例"（bookmark-api 的分页方案变更），但那是单模块内的局部变更。中型项目的需求变更可能影响多个模块：

- "增加优惠券功能"→ 影响 order、payment、user 三个模块的数据模型、接口、测试
- "把用户认证从密码改为 OAuth"→ 影响所有需要鉴权的接口

当前流程没有跨模块影响分析的机制——变更了 api-contracts.md 的某个接口后，如何系统性地识别所有受影响的测试和实现任务？靠人工在 10+ 模块间逐个排查是不现实的。

**崩溃点 14：文档腐化**

小项目规模小，文档基本不会过时。中型项目模块多、迭代多，architecture.md、api-contracts.md 会随着实现过程中的发现而需要多次修改。当前流程虽然有"文档优先"的原则，但没有文档与实现一致性的检查机制。随着时间推移，文档和实际代码之间的偏差会逐渐积累，后续 agent 读到的文档可能已经不能准确反映代码现状。

**崩溃点 15：技术债的累积和管理**

小项目规模小，技术债还来不及累积就交付了。中型项目开发过程中，为了赶进度或绕过难题，会产生需要后续修复的技术债（如硬编码的配置、缺失的错误处理、权宜的数据结构）。当前流程没有技术债记录和清偿的机制。

## 4. 代码量和认知负荷

### 4.1 量变

| 维度 | 小型项目 | 中型项目 |
|------|---------|---------|
| 总代码量 | 1K-5K 行 | 10K-50K 行 |
| 单模块代码量 | 200-500 行 | 500-3000 行 |
| 文件数 | 10-30 个 | 50-200 个 |
| 测试用例数 | 10-30 个 | 60-300 个 |
| 文档总量 | 200-500 行 | 1000-3000 行 |

### 4.2 小项目流程在此崩溃的具体点

**崩溃点 16：单个 agent 无法在一个会话中理解整个项目**

小项目的全部代码 + 文档 + 测试可以在一个 agent 会话的上下文窗口中完整装下（约 5K-15K 行，对于 200K token 的上下文绰绰有余）。中型项目的代码 + 文档 + 测试总量可能达到 50K-100K 行，即使用 1M token 上下文也无法全部装入，更不用说 agent 还需要留出空间做推理。

这意味着：
- agent 无法"通读全部代码后理解全貌"，必须依赖文档做间接理解
- 修改一个模块时，agent 可能不知道另一个模块中存在依赖于被修改行为的代码
- "不要做的事"列表依赖 agent 对全局的理解，但 agent 看不到全局

**崩溃点 17：api-contracts.md 单文件无法承载**

小项目的接口契约文件通常 100-300 行（bookmark-api 有 10 个接口）。中型项目可能有 40-80 个接口，接口契约文件膨胀到 1000-3000 行。单文件难以阅读、难以定位、容易产生编辑冲突。Tester agent 只关心其中 3-5 个接口的契约，却要在上下文中装载全部 3000 行。

**崩溃点 18：测试执行时间变长**

小项目 10-30 个测试用例，全量执行 < 10 秒。中型项目 60-300 个测试用例，全量执行可能需要 30 秒到几分钟。agent 实现一个模块后跑全量测试的反馈循环变长。更严重的是，其他模块的测试失败可能干扰当前模块的开发（agent 可能误以为自己的修改导致了其他模块的测试失败，陷入错误的修复方向）。

**崩溃点 19：task 粒度判断变难**

小项目的 task 拆分直观——一个模块就是一个实现任务（如 Impl-001: 实现 JSON 文件读写）。中型项目的单个模块可能有 2000-3000 行代码，超出了 talk.txt 中建议的"改动行数 < 500 行"的 task 粒度。需要将一个模块进一步拆分为多个 task，但如何拆、拆到什么粒度，当前流程没有指导。

**崩溃点 20：PRD 单文件不可行**

小项目的 PRD 50-100 行。中型项目 4-10 个业务模块的完整需求描述可能达到 500-1500 行。根据 talk.txt 中的判断标准，500+ 行就需要考虑拆文件。但当前 PRD 模板是单文件结构，没有拆分和跨文件引用的规范。

## 5. 典型中型项目举例

### 5.1 团队协作平台（类简化版 Notion）

- 模块：用户、工作空间、页面管理、权限控制、实时编辑、评论、通知、搜索
- 为什么不适合小项目流程：8 个模块间存在复杂权限依赖（工作空间→页面→评论都有权限继承），实时编辑涉及并发状态管理，通知模块需要监听几乎所有其他模块的事件

### 5.2 电商后台管理系统

- 模块：用户、商品、分类、订单、支付、库存、优惠券、数据统计
- 为什么不适合小项目流程：下单是跨 4 个模块的编排操作（订单→库存→支付→优惠券），库存是共享状态热点，需求变更频繁（促销规则每月调整），数据统计依赖所有业务模块的数据

### 5.3 内容管理系统（CMS）

- 模块：用户/角色、内容管理、媒体库、分类/标签、审核流程、发布/版本管理、评论、SEO 配置
- 为什么不适合小项目流程：审核流程涉及状态机（草稿→待审→通过→发布→下架），版本管理引入内容快照和回滚机制，媒体库需要与内容管理深度集成，角色权限贯穿所有模块

### 5.4 项目管理工具（类简化版 Jira）

- 模块：用户、项目、任务/子任务、看板/泳道、Sprint 管理、评论/活动流、通知、报表
- 为什么不适合小项目流程：任务有复杂的状态流转和依赖关系，看板视图需要聚合多模块数据，活动流需要记录所有模块的变更事件，Sprint 管理涉及跨项目资源分配

### 5.5 多租户 SaaS API 平台

- 模块：租户管理、用户/认证、API Key 管理、用量计量、计费/订阅、配额控制、审计日志、Webhook
- 为什么不适合小项目流程：每个请求都需要经过租户隔离→认证→配额检查→计量的中间件管道，计费和配额涉及跨模块的共享状态，审计日志需要拦截所有模块的操作

### 5.6 在线教育平台后端

- 模块：用户/角色（学生/教师/管理员）、课程管理、课程内容/章节、作业/测验、成绩管理、支付/订阅、通知、进度追踪
- 为什么不适合小项目流程：三种用户角色对同一资源（课程）有不同的操作权限和视图，作业→批改→成绩是跨模块的业务流程，进度追踪需要监听内容浏览和作业提交事件

### 5.7 预约/排班系统

- 模块：用户、服务项目管理、员工/资源管理、排班/可用时段、预约管理、支付、通知/提醒、评价
- 为什么不适合小项目流程：预约创建涉及可用时段锁定（并发竞争问题），排班和预约有复杂的时间冲突检测逻辑，取消/改约涉及退款和时段释放的事务性操作

|--------|--------|------|---------|
| P0 | #16 agent 无法理解整个项目 | 认知负荷 | 第一个 agent 开始实现时 |
| P0 | #10 Git 并行工作流缺失 | 团队协作 | 第二个人开始并行开发时 |
| P0 | #7 task-board 并发瓶颈 | 团队协作 | 多人并行开发时 |
| P0 | #12 没有迭代概念 | 迭代与变更 | 项目规划阶段 |
| P1 | #17 api-contracts 单文件溢出 | 认知负荷 | 架构设计阶段 |
| P1 | #20 PRD 单文件溢出 | 认知负荷 | 需求编写阶段 |
| P1 | #5 跨模块事件级联无机制 | 模块关系 | 模块间开始交互时 |
| P1 | #2 循环依赖检测缺失 | 模块关系 | 模块数 > 5 后 |
| P1 | #8 CLAUDE.md 无法承载差异化规则 | 团队协作 | 模块有不同规范时 |
| P1 | #19 task 粒度拆分无指导 | 认知负荷 | 任务规划阶段 |

| P2 | #4 多对多关系归属困难 | 模块关系 | 多对多关系 > 3 时 |
| P2 | #6 共享状态一致性 | 模块关系 | 有并发写入场景时 |

| P2 | #18 测试执行时间变长 | 认知负荷 | 测试用例 > 100 后 |

## 7. 核心结论

小项目流程的三个根本假设在中型项目中不再成立：

1. **"一个 agent 能理解全局"假设** — 小项目的全部代码和文档能装入单个 agent 上下文。中型项目不行。这直接摧毁了当前流程中 agent 自主做出正确决策的基础。

2. **"串行执行"假设** — 小项目可以一个人一个任务接一个任务地做。中型项目必须多人并行，而当前流程的 Git 策略、task-board 更新机制、review 流程都建立在串行假设上。

3. **"一次性交付"假设** — 小项目从头做到尾，中间不发版、不变更需求。中型项目必须分迭代、能响应需求变更，而当前流程没有迭代管理和变更传播的机制。

中型项目的流程设计需要围绕打破这三个假设来重新构建。
