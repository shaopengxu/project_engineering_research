# PRD 编写指引（中型项目）

本指引帮助产品经理编写满足中型项目开发流程要求的 PRD。AI Agent 将基于 PRD 进行架构设计、模块拆分和任务规划，因此 PRD 的结构和完整性直接影响开发质量。

## PRD 最低要求

技术负责人在 Step 1 审查 PRD 时，需确认以下条目全部满足：

```
- [ ] 项目概述清晰（解决什么问题、目标用户是谁）
- [ ] 功能点按业务模块组织（不是按页面或按流程零散罗列）
- [ ] 每个功能点有明确的验收标准（可测试、无二义性）
- [ ] 用户角色和权限已定义
- [ ] 非功能性需求已说明（性能、安全、兼容性等，至少写"无特殊要求"）
- [ ] 明确标注"不在范围内"的功能（防止范围蔓延）
- [ ] 业务规则具体（不使用"合理的"、"适当的"等模糊用语）
```

```markdown

{为什么要做这个项目，解决什么问题}

### 1.2 目标用户
{谁会使用这个系统，列出所有用户角色}

### 1.3 核心价值
{一句话说明这个项目的核心价值}

| 角色 | 描述 | 权限范围 |

## 3. 功能需求

> 按业务模块组织。每个模块对应系统的一个独立业务领域。

{这个模块做什么，包含哪些核心功能}

- 描述：{具体做什么}

  - {规则1：具体、可验证、无二义性}

  - [ ] {标准1：描述预期行为和结果}

  - {规则1}

  - [ ] {标准1}

### 3.2 {业务模块B}：{模块名称}

{同上格式}

### 3.N {业务模块N}

{同上格式}

## 4. 非功能性需求

### 4.1 性能要求
- {如：API 响应时间 < 500ms（P95）}
- {如：首页加载时间 < 3s}
- {或：无特殊性能要求}

### 4.2 安全要求
- {如：密码必须加密存储}

- {或：遵循基本安全实践即可}

- {如：支持 Chrome/Firefox/Safari 最新两个版本}

- {不做的功能3}
```

## 验收标准编写规范

好的验收标准应当满足：

- **可测试**：能通过自动化测试或手动操作验证
- **无二义性**：不同人阅读得到相同理解
- **具体**：包含具体的输入、操作和预期结果

| 示例类型 | 不好的写法 | 好的写法 |
|---------|-----------|---------|
| 输入校验 | 用户名需要合理长度 | 用户名长度 2-20 个字符，仅允许字母、数字、下划线 |
| 状态流转 | 订单可以取消 | 订单状态为"待支付"时可取消，"已发货"后不可取消 |
| 权限控制 | 管理员有更多权限 | 管理员可以查看所有用户的订单，普通用户只能查看自己的 |
| 业务规则 | 库存不足时要提示 | 下单时库存不足，返回错误码 INSUFFICIENT_STOCK，不创建订单 |

## 业务规则编写规范

业务规则是 Agent 实现功能和编写测试的核心依据，必须具体明确：

- 避免使用"合理的"、"适当的"、"必要时"等模糊用语
- 数值类规则给出具体数字或范围
- 条件分支逻辑列出所有分支的处理方式
- 涉及多个模块的规则，在相关模块中都要提及

## PRD 拆分建议

当 PRD 超过 500 行时，建议拆分为目录结构：

```
docs/
├── prd/
│   ├── overview.md          # 项目概述 + 用户角色 + 非功能性需求 + 不在范围内
│   ├── module-a.md          # 模块A 功能需求
│   ├── module-b.md          # 模块B 功能需求
│   └── ...
```
